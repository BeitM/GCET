# RoboLab FTC

> AI-powered virtual simulation and debugging feedback for FIRST Tech Challenge programmers.

RoboLab FTC combines virtual robot simulation, telemetry, and AI-guided explanation in one learning environment. The platform is designed to help FTC programmers understand not only **what** their robot did, but **why** it behaved that way and **how** to improve the code.

## Overview

FTC programmers write software that controls physical systems. As a result, code errors often appear as unexpected robot behavior instead of clear error messages. A small mistake can affect movement, mechanism behavior, timing, safety, or overall routine performance.

RoboLab FTC is intended to make the connection between code and physical behavior easier to understand. Students will be able to test robot programs in a virtual FTC environment, inspect telemetry from the simulation, and work with an AI mentor that explains the results in context.

The central product workflow is:

**Goal + Code + Virtual Simulation + Telemetry → AI Feedback + Debugging and Optimization Guidance**

## The Problem

### Code errors become physical robot behavior

Beginner programmers often struggle to understand why a robot behaves differently from what the code appears to describe. The visible symptom may be unexpected or inconsistent physical behavior rather than an obvious software error.

### Existing tools are not fully diagnostic

FTC simulation tools can demonstrate what a virtual robot does, but they do not always explain why it happened. Telemetry provides detailed measurements of robot state and performance, but students may not know how to connect that data back to the relevant code.

This creates a gap between observing a problem and understanding how to fix it.

## The Solution

RoboLab FTC adds an AI mentor layer to virtual FTC robot simulation.

The system is planned to analyze:

- the behavior the student intended,
- the robot code,
- the robot configuration,
- telemetry generated by the simulation, and
- the robot's actual movement and subsystem behavior.

Using that context, the AI mentor can explain what happened, identify likely causes, reference supporting telemetry, recommend debugging steps, and suggest performance improvements.

The platform is intended to support both outcomes:

- **Debugging:** explain mismatches, errors, unsafe behavior, and failed routines.
- **Optimization:** identify more efficient behavior, better coordination, and greater consistency when the routine already works.

## How It Works

### 1. The student provides context

The student enters:

- the intended robot behavior or goal,
- the code they want to test, and
- information about the robot and its subsystems.

For example:

> Complete an autonomous field objective while coordinating movement and robot mechanisms reliably.

### 2. The robot runs in simulation

A virtual robot executes the routine in an FTC-style field environment. The simulation represents robot movement, interaction with the field, and the behavior of its configured mechanisms.

### 3. The simulation generates telemetry

The platform records telemetry describing:

- robot state and movement,
- control outputs and feedback,
- mechanism state and coordination, and
- sensor and event data.

### 4. The AI mentor analyzes performance

The AI compares the intended behavior with the code and observed simulation results. It looks for mismatches, inefficient execution, unsafe or inconsistent behavior, and opportunities for improvement.

### 5. The student receives actionable feedback

RoboLab FTC presents the diagnosis in structured sections:

- **What happened** - a plain-language description of the observed behavior
- **Likely cause** - the code, configuration, or control issue that may be responsible
- **Telemetry evidence** - measurements and events that support the diagnosis
- **Suggested fix** - concrete debugging steps
- **Optimization idea** - ways to improve an already functional routine
- **Concept explanation** - an accessible explanation of the underlying robotics concept

## Core Capabilities

The planned platform includes:

- A virtual FTC-style field and simulated robot
- A preset robot for the first prototype
- Future support for custom robot configurations
- Safe testing of FTC robot code and autonomous routines
- Live telemetry describing robot state, control feedback, and mechanism behavior
- Visual representations of movement and configured robot mechanisms
- AI feedback based on code, telemetry, and simulation behavior
- Guided demonstration scenarios for learning and testing
- Debug reports that summarize evidence and recommended changes

## Example Use Cases

### Autonomous routine optimization

A robot completes an autonomous objective but uses unnecessary movement or delays. RoboLab FTC could recognize that the routine works while recommending a more efficient sequence and better coordination between actions.

### Mechanism timing issue

A mechanism activates before another required action is complete. RoboLab FTC could use telemetry to identify the timing mismatch, explain its effect on the routine, and recommend a state-based readiness check instead of a fixed delay.

### Mechanism safety problem

A mechanism continues moving beyond its intended operating range. The simulation could expose the unsafe behavior while the AI mentor identifies missing feedback, stopping conditions, or safety limits.

### Drivetrain inconsistency

A robot is commanded to follow a path but gradually deviates from it. RoboLab FTC could compare commanded and observed motion to explain the difference between open-loop commands and feedback-based control.

## Why It Matters

FTC programming requires students to connect abstract software with real mechanical and electrical behavior. RoboLab FTC is intended to make that relationship more visible and teachable.

For beginner programmers, the platform can reduce time spent guessing and provide explanations that build transferable robotics knowledge. For experienced teams, it can support autonomous optimization, subsystem coordination, and testing before risking physical hardware.

Expected impact includes:

- helping students learn FTC programming faster,
- supporting teams without experienced programming mentors,
- making telemetry easier to interpret,
- reducing trial-and-error debugging,
- encouraging safer testing before using a physical robot, and
- turning simulation into an interactive learning experience.

## Simulation Foundation

Existing tools such as [Virtual Robot Simulator](https://github.com/Virtual-FTC) demonstrate that FTC robot testing in a virtual environment is possible.

RoboLab FTC's focus is the educational layer around simulation: combining a student's goal, code, telemetry, and observed behavior to produce tailored explanations and debugging guidance. The project may integrate with or learn from established FTC simulation tools instead of recreating every simulation capability independently.

## Prototype Plan

The first version is intended to prove the core idea of AI-assisted feedback inside a virtual FTC-style environment.

Essential prototype capabilities are:

1. A virtual FTC-style field
2. A preset simulated robot
3. Support for key robot subsystems
4. Inputs for code, robot setup, and intended behavior
5. A telemetry dashboard
6. An AI assistant that explains bugs, mismatches, and optimization opportunities
7. Guided example scenarios that demonstrate the workflow

The first prototype does not need full FTC SDK compilation, perfect physics, CAD import, real-robot connectivity, user accounts, or persistent project storage. Those capabilities can follow after the central learning workflow has been validated.

## Current Repository Status

This repository contains a working simulator-first prototype of the planned experience. It demonstrates the core end-to-end workflow but is not the completed product described above.

The current implementation includes:

- A Next.js landing page with separate Sandbox and structured Learning Mode entry paths
- A shared simulator dashboard that exposes the full setup in Sandbox and level-based guidance in Learning Mode
- An interactive 3D DECODE field with Rapier artifact and shot physics
- One preset turret-shooter robot, configurable start pose, alliance, preload, and artifact rows
- A small RoboLab autonomous command DSL, including JavaScript-style per-motor `setPower` calls and open-loop mecanum motion, plus keyboard TeleOp controls
- Telemetry recording, timeline playback, rule warnings, and classifier scoring
- Deterministic mentor feedback with optional OpenAI-generated analysis and follow-up chat
- A validated and bounded `/api/analyze` route that receives code, robot setup, and compact telemetry

Current limitations:

- The code editor executes supported function-call statements, not arbitrary JavaScript or FTC SDK Java.
- Physics and DECODE rules are partial rather than competition-accurate.
- Only one robot preset and one season are active.
- CAD import, custom robot construction, accounts, persistence, and team workflows are placeholders.
- There is no automated test suite or CI configuration yet.

These constraints reflect the current implementation stage, not the intended final scope of RoboLab FTC.

## Getting Started

### Prerequisites

- Node.js 20 or newer
- `pnpm` through Corepack or a global pnpm installation

### Install and run

```bash
git clone https://github.com/BeitM/robolab.git
cd robolab/my-app/frontend
corepack enable
pnpm install
pnpm dev
```

Open [http://localhost:3000](http://localhost:3000) to choose a mode. The open Sandbox is available at [http://localhost:3000/simulator?mode=sandbox](http://localhost:3000/simulator?mode=sandbox), and the structured level hub is at [http://localhost:3000/learn](http://localhost:3000/learn).

### Scripts

Run commands from `my-app/frontend`:

| Command | Purpose |
| --- | --- |
| `pnpm dev` | Start the development server |
| `pnpm build` | Create and type-check a production build |
| `pnpm start` | Run the production build |
| `pnpm lint` | Run ESLint |
| `pnpm typecheck` | Run TypeScript without emitting files |

## Development Principles

- Keep the educational feedback layer central to product decisions.
- Treat the official FIRST competition manual as authoritative for rules and field specifications.
- Keep simulation, telemetry, and AI analysis behind replaceable interfaces.
- Distinguish clearly between implemented features and planned capabilities.
- Validate the experience with FTC students and mentors as early as possible.
